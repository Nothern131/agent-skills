# Agent Skills — 技术方案

## 系统架构

### 整体架构

```
┌─────────────────────────────────────────────────┐
│                  用户 / 开发者                     │
└──────────────────────┬──────────────────────────┘
                       │ 编码任务
                       ▼
┌─────────────────────────────────────────────────┐
│            AI 编码助手（任意模型）                  │
│  ┌─────────────────────────────────────────────┐ │
│  │         model-upgrade 框架（四层防御）         │ │
│  │  Layer 1: 铁律规则 — 15条工程纪律             │ │
│  │  Layer 2: 项目感知 — PROJECT_CONTEXT.md       │ │
│  │  Layer 3: 任务拆解 — 大→小+检查点             │ │
│  │  Layer 4: 测试修正 — 生成→验证→修正循环        │ │
│  └─────────────────────────────────────────────┘ │
│  ┌──────────────┐  ┌──────────────────────────┐  │
│  │ clarifying-  │  │      codeguard           │  │
│  │ questions    │  │  80规则 × 8维度 × 5语言    │  │
│  │ 需求反问系统  │  │  0-100评分 + 修复建议      │  │
│  └──────────────┘  └──────────────────────────┘  │
└──────────────────────┬──────────────────────────┘
                       │ 生产级代码
                       ▼
┌─────────────────────────────────────────────────┐
│              代码仓库 / 生产环境                    │
└─────────────────────────────────────────────────┘
```

### 数据流

```
用户需求
  │
  ├─→ clarifying-questions（需求澄清）
  │     └─→ 明确的需求描述
  │
  ├─→ model-upgrade Layer 1（铁律检查）
  │     └─→ 影响分析 + 约束列表
  │
  ├─→ model-upgrade Layer 2（上下文注入）
  │     └─→ PROJECT_CONTEXT.md 加载
  │
  ├─→ model-upgrade Layer 3（任务拆解）
  │     └─→ 步骤列表 + 检查点定义
  │
  └─→ 执行循环（每步）：
        ├─→ 生成/修改代码
        ├─→ codeguard 扫描（语法+语义）
        ├─→ 自修正（最多3次）
        ├─→ 运行测试
        ├─→ 自修正（最多3次）
        └─→ 检查点验证 → 下一步
```

## 核心模块详细设计

### 模块一：model-upgrade（模型升级框架）

#### 四层防御体系

**Layer 1：铁律规则（Rules）**

15 条不可违反的工程纪律，按优先级排列：

| # | 规则 | 解决的差距 | 违反后果 |
|---|------|-----------|---------|
| 1 | 影响分析先行 | 指令跳跃 | 改错文件、漏改依赖 |
| 2 | 单一关注点 | 多任务混乱 | 改动互相干扰 |
| 3 | 禁止静默失败 | 错误隐藏 | 线上事故无法定位 |
| 4 | 公共接口文档化 | 信息丢失 | 接口不一致 |
| 5 | 测试驱动变更 | 质量不可验证 | 回归bug |
| 6 | 自动验证 | 跳过检查 | 语法错误上线 |
| 7 | 回归预防 | 修A坏B | 老bug复发 |
| 8 | 禁止硬编码密钥 | 安全盲区 | 密钥泄露 |
| 9 | 输入验证 | 安全漏洞 | 注入攻击 |
| 10 | 参数化查询 | SQL注入 | 数据泄露 |
| 11 | 检查点验证 | 步骤遗漏 | 任务不完整 |
| 12 | 三次尝试上限 | 无限循环 | 浪费token |
| 13 | 上下文维护 | 上下文丢失 | 后续任务脱节 |
| 14 | 展示工作 | 不透明 | 用户无法审查 |
| 15 | 总结变更 | 信息丢失 | 变更不可追溯 |

**Layer 2：项目感知（Context Injection）**

PROJECT_CONTEXT.md 维护项目的全局视野：

```markdown
# PROJECT_CONTEXT.md
## 目录结构（相关部分）
## 关键实体关系
## 关键配置
## 最近变更
## 依赖与契约
## 已知问题
## 测试策略
## 构建与部署
```

工作原理：
1. 会话开始时生成/加载 PROJECT_CONTEXT.md
2. 每个任务执行前注入相关上下文
3. 任务完成后更新上下文

**Layer 3：任务拆解（Task Decomposition）**

三种拆解模式：

| 模式 | 适用场景 | 方法 |
|------|---------|------|
| A: 大→小步骤 | 新功能、重构 | 按依赖关系拆分，每步可独立验证 |
| B: 填空模板 | 复杂实现 | 提供代码骨架，模型填充细节 |
| C: 检查点执行 | 长任务 | 每步完成后自检，通过才继续 |

**Layer 4：测试修正循环（Test-Feedback Loop）**

```
生成代码 → 语法检查 → (失败→自修正×3) → 单元测试 → (失败→读错误→修正×3) → 完成
```

关键设计：
- 语法检查优先于测试（快速反馈）
- 错误信息是修正的关键输入（不是盲目重试）
- 三次尝试上限（避免无限循环浪费token）
- 根因修复而非表面修补

#### 提示词模板

4 套即用模板，针对最常见的编码场景：

1. **复杂重构**：理解→计划→执行→验证→文档
2. **Bug修复**：复现→追踪→修复→验证→回归测试
3. **新功能**：模型→契约→测试→实现→验证
4. **代码审查**：安全→错误处理→性能→可读性→最佳实践

### 模块二：codeguard（代码质量扫描器）

#### 8维度80规则体系

| 维度 | ID范围 | 规则数 | 核心关注 |
|------|--------|--------|---------|
| 错误处理 | ERR001-010 | 10 | try-catch缺失、空catch、未处理Promise |
| 安全 | PERM001-010 | 10 | 硬编码密钥、注入风险、认证缺失 |
| 数据库 | DB001-010 | 10 | 缺事务、连接泄漏、N+1查询 |
| 诊断 | DIAG001-010 | 10 | 缺日志、无错误码、无追踪ID |
| 韧性 | RES001-010 | 10 | 无重试、无熔断、无超时 |
| 内存安全 | MEM001-010 | 10 | 缓冲区溢出、use-after-free、RAII |
| 并发 | CON001-010 | 10 | 竞态条件、死锁、GDScript deferred |
| 资源管理 | RES_M001-010 | 10 | 文件泄漏、无界缓存、GDScript节点生命周期 |

#### 评分算法

```
score = 100 - (total_deduction / max_possible_deduction) × 100
```

| 严重度 | 扣分 | 示例 |
|--------|------|------|
| Critical | 40 | 缓冲区溢出、SQL注入 |
| High | 25 | 空catch、竞态条件 |
| Medium | 15 | 缺日志、无超时 |
| Low | 5 | 缺docstring |

#### 规则结构

每条规则包含：

```
1. Rule Table（ID、严重度、检测描述）
2. Detection Patterns（违规代码模式，多语言）
3. Fix Examples（修复代码，多语言）
```

### 模块三：clarifying-questions（需求反问系统）

#### 工作原理

```
用户模糊需求 → 识别模糊点 → 生成针对性问题 → 用户回答 → 明确需求
```

#### 反问策略

| 策略 | 适用场景 | 示例 |
|------|---------|------|
| 范围澄清 | 需求边界不清 | "这个功能需要支持哪些用户角色？" |
| 技术选型 | 实现方式未定 | "你倾向于用 REST 还是 GraphQL？" |
| 优先级确认 | 多个方向可选 | "性能和可读性，哪个优先？" |
| 约束发现 | 隐含约束未说明 | "这个系统需要支持多少并发用户？" |

## 技术选型

| 决策 | 选择 | 理由 |
|------|------|------|
| 规范格式 | Agent Skills Spec | 业界标准，10+工具兼容 |
| 规则存储 | Markdown references/ | 人类可读，按需加载 |
| 评分方式 | 扣分制 | 直观，易于理解 |
| 上下文管理 | PROJECT_CONTEXT.md | 单一真相源，自动维护 |
| 语言覆盖 | Python/JS/C++/GDScript/Go | 覆盖主流开发场景 |

## 性能指标

| 指标 | 目标 | 实现方式 |
|------|------|---------|
| 代码质量提升 | 30%+ | codeguard 80规则扫描 |
| 任务完成率 | 95%+ | 四层防御+三次尝试上限 |
| 回归bug率 | <5% | 测试驱动+回归测试 |
| 安全漏洞 | 0 Critical | PERM规则+输入验证 |
| 内存安全 | 0 Critical (C/C++) | MEM规则+RAII强制 |
